# remote_process_control 软件功能说明书

版本：V1.0（基于仓库当前代码快照生成）  
适用范围：`remote_process_control` 全工程（前端 / 信令服务 / Windows Agent / 独立接收端）  

---

## 1. 文档目的

本文档用于说明 `remote_process_control` 工程在当前仓库版本下提供的功能、组成模块、端到端业务流程、对外协议（信令与 DataChannel）、关键配置项、异常边界与验收要点，供研发、测试、运维与交付使用。

---

## 2. 产品概述

`remote_process_control` 是一套基于 WebRTC 的远程进程控制与文件传输系统，核心能力包括：

- **远程进程画面与控制**：前端发起会话请求，远端 Windows Agent 启动目标进程并采集窗口画面，编码 H264 经 WebRTC 视频轨下发；鼠标/键盘通过 DataChannel 上行，远端注入输入。
- **“我的数据”文件管理**：前端通过 DataChannel 浏览远端固定根目录（沙箱），支持文本预览、上传、下载；大文件采用分片、ACK 与重试。
- **可观测性与稳定性**：内置延迟诊断（DataChannel 对时 / WebRTC stats），支持关键帧请求用于丢包恢复；会话断线提供宽限期处理。

## 3. 系统组成与部署形态

### 3.1 组件清单

- **前端（浏览器）**：`frontend/`  
  入口：`frontend/index.html`，桌面容器与“我的数据”：`desktopMode.js`、`my_data.html`、`my_data.js`；WebRTC 与信令：`webrtcLayer.js`、`signalingLayer.js`。

- **信令服务（Qt WebSocket 中继）**：`signaling-server-qt/`  
  入口：`signaling-server-qt/main.cpp`，默认监听 `0.0.0.0:9090`。

- **Windows 远端 Agent（Offerer/发送端）**：`remote_desktop_agent/`  
  入口：`remote_desktop_agent/app/main.cpp`，负责会话编排、进程拉起、采集/编码/发送、输入注入、文件传输。

- **独立接收端（可选，Answerer/接收端）**：`remote_desktop_receiver/`  
  入口：`remote_desktop_receiver/src/main.cpp`，用于非浏览器场景的解码呈现、输入回传与性能验证。

### 3.2 总体通信拓扑

```
前端（浏览器） <--WebSocket 信令--> Qt 信令服务 <--WebSocket 信令--> Windows Agent
      ^                                                          |
      +------------------- WebRTC(视频轨 + DataChannel) ----------+
```

说明：
- 信令层**仅转发**文本 JSON（按 id 路由），不承载媒体。
- 媒体、输入、文件传输等均通过 WebRTC DataChannel/Track 完成。

## 5. 功能需求说明（按模块）

## 5.1 前端（`frontend/`）

### 5.1.1 连接与会话控制
- 支持输入/配置信令地址（URL 参数 `signaling=` 或默认推导）。
- 发起会话请求：
  - 媒体会话：`type=request`，可携带 `exePath`。
  - 仅文件会话：`type=file_request`（若需要）。
- 处理 Agent 发来的 SDP Offer，生成 Answer 并回传。
- 展示 WebRTC 连接状态（ICE/信令/连接状态等）。

### 5.1.2 远程画面显示
- 接收视频轨并播放到主视频元素。
- 处理分辨率变化：更新 UI 显示与鼠标映射参数。
- 断开/失败处理：
  - 连接失败或长时间 disconnected 可触发自动退出/关页（窗口模式）。

### 5.1.3 远程控制（DataChannel 上行）
- 建立 DataChannel 后，发送 `controlRequest` 请求控制权。
- 在控制权授予后，上报鼠标/键盘事件：
  - 鼠标移动支持绝对坐标与视频分辨率字段，用于远端精确映射。
  - 支持滚轮、双击等常用操作。

### 5.1.4 延迟/质量诊断（可观测性）
- DataChannel `latPing/latPong` 对时与 RTT 估算，输出诊断日志。
- 周期 `getStats`：
  - 统计抖动缓冲、解码耗时、ICE RTT、丢帧等指标。
  - 在抖动/丢包严重时发送 `requestKeyframe` 请求关键帧恢复。

### 5.1.5 “我的数据”文件管理
- 浏览目录：获取目录列表并渲染（支持进入子目录/返回上级）。
- 文本预览：对可预览类型请求内容（有最大字节上限）。
- 下载：按 `transferId` + `offset` 拉取 base64 分片并在浏览器端合并为 Blob 下载。
- 上传：按分片 base64 上送，等待 ACK、超时重试与偏移对齐；完成后提交 commit。

## 5.2 信令服务（`signaling-server-qt/`）

### 5.2.1 基本功能
- 监听 WebSocket（默认 `0.0.0.0:9090`）。
- 连接身份：通过 URL path `/<client_id>` 获取客户端 id 并登记。
- 消息中继：
  - 从来源收到 JSON 文本后，读取其中 `id` 作为目标 id。
  - 若目标存在，则转发给目标，并将转发消息的 `id` 改写为**来源 id**（便于对端识别发送者）。
- 不解析 SDP/ICE 语义，不持久化会话状态，不做鉴权（当前版本）。

## 5.3 Windows 远端 Agent（`remote_desktop_agent/`）

### 5.3.1 应用启动与配置
- 入口程序初始化 DPI 感知，读取运行时配置，连接信令服务并进入阻塞运行（等待退出输入）。

### 5.3.2 信令接入与事件化
- WebSocket 连接到 `ws://<ip>:<port>/<signaling_local_id>`。
- 将入站 JSON 转为领域事件 `signaling_event`：
  - `request` -> 媒体会话请求（可带 exePath）
  - `file_request` -> 仅文件会话请求
  - `answer` -> SDP Answer
  - `stop` -> 停止会话
- 通过内部队列串行派发，避免 WebSocket 回调线程与业务线程交织。

### 5.3.3 会话编排（单会话模型）
- 全局同时最多一个活动会话 `active_desktop_session`。
- 收到新请求默认“拆旧建新”（可扩展替换策略）。
- 断线处理：
  - operator 连接丢失后进入宽限期（当前实现为 60 秒），到期仍未恢复则 teardown。
- 会话 teardown 幂等：
  - 广播 `remoteProcessExited`（用于前端/接收端退出流程）
  - 停止媒体、解绑输入、关闭 PeerConnection、释放文件服务与资源。

### 5.3.4 WebRTC（Offerer）与 DataChannel
- 构建 PeerConnection，创建音视频轨与 DataChannel。
- DataChannel 支持：
  - 心跳：纯文本 `Ping/Pong`
  - 控制权：`controlRequest/controlRelease` 与 `controlGranted/controlDenied/controlRevoked`
  - 输入注入：鼠标/键盘事件
  - 关键帧：`requestKeyframe`
  - 延迟对时：`latPing/latPong`
  - 文件协议：fileList/filePreview/fileDownload*/fileUpload*

### 5.3.5 输入注入（键鼠）
- 通过 `SendInput` 注入鼠标/键盘。
- 坐标映射策略（优先级从高到低）：
  - 使用采集线程提供的**合并采集矩形**进行映射（`set_capture_screen_rect`）。
  - 若已绑定目标窗口（主窗口 hwnd），使用窗口有效矩形映射。
  - 退化为屏幕分辨率比例映射或相对移动。
- 注入前置前台：
  - 尝试将目标窗口置前台并恢复显示，降低系统前台锁影响。

### 5.3.6 媒体采集与编码（视频）

#### 5.3.6.1 采集目标解析（窗口/进程选择）
- `CaptureTargetResolver` 在每帧循环中解析采集目标，核心思路：
  - 优先使用会话 PID 集合（Job identity）合并枚举 surfaces。
  - 若已有可用主窗口 hint，优先沿用。
  - 按当前 capture_pid 枚举可见顶层窗口作为快路径。
  - 恢复路径：允许 rebind 时，按 `launch_pid` 的后代进程树与 `exe basename` 进行窗口恢复。
- `WindowScorePolicy` 对窗口做可见性、风格、面积、splash token 等评分，辅助选择“最合适主窗口”。

#### 5.3.6.2 采集后端与管线
- 采集后端：
  - DXGI Desktop Duplication（优先，支持多显示器选择输出器）
  - GDI（回退）
- 管线：
  - 多窗口 tiles 裁剪与合成（FrameComposer）
  - 可疑帧过滤（FrameFilter，例如黑帧过滤）
  - 统一编排（CapturePipeline）

#### 5.3.6.3 线程模型与数据流（采集 -> 编码 -> 发送）
- `CaptureWorker`（采集线程）：
  - 按 FPS 节拍循环，解析目标、grab raw frame，写入“最新帧槽位（LatestRawFrame）”，唤醒编码线程。
  - 无 surfaces 时进入 grace（当前为 5s）：进程仍活则广播 `remoteWindowMissing`（不退出），进程不活则触发退出通知。
- `EncodeWorker`（编码线程）：
  - 从“最新帧槽位”取到新 `frame_id` 后编码 H264。
  - 编码结果写入“最新编码队列（BoundedQueue，默认最多 3 帧；溢出丢最旧）”。
- 样本消费（发送调度）：
  - `produce_next_video_sample()` 无阻塞取走最新编码样本，交给 sender 发送到 video track。
  - 编码失败时可触发 `request_force_keyframe()` 促进恢复。

#### 5.3.6.4 进程生命周期与 Job identity
- `ProcessSession` 拉起进程后优先创建 Job 并分配 launch 进程，使“会话 PID 集合”稳定可追踪（避免子进程/父链变化导致目标丢失）。
- `session_pids()`：若有 job identity 则查询 Job 的进程 id 列表，否则退化为 `{capture_pid, launch_pid}`。

### 5.3.7 文件传输（DataChannel）

#### 5.3.7.1 能力范围
- 目录列表（list）
- 文本文件预览（preview，限定扩展名与字节上限）
- 下载（init + 按 offset 拉取分片）
- 上传（init + 分片 + commit，支持 resume 语义与偏移对齐）

#### 5.3.7.2 路径安全与沙箱
- 文件根目录由 `RPC_DATA_ROOT` 限定。
- 禁止绝对路径，禁止通过 `..` 等方式逃逸根目录。
- 上传 fileName 禁止包含路径成分（只允许文件名）。

#### 5.3.7.3 分片与编码
- 二进制分片通过 JSON + base64 承载。
- 后端对 chunk 大小进行钳制（最小 4KB，最大 64KB；若未提供则使用默认值再钳制）。

## 5.4 独立接收端（`remote_desktop_receiver/`，可选）

### 5.4.1 用途
- 在非浏览器环境下接收 WebRTC 视频轨并解码呈现（D3D），用于验证/演示/性能测试。

### 5.4.2 主要功能
- 命令行参数：
  - `--host/--port/--clientId/--exePath`
  - `--decoder auto|hw(d3d11va)|sw`
  - `--windowed/--fullscreen/--allMonitors`
  - `--maxFps`
- WebRTC Answerer：接收 Offer，生成 Answer 并回传。
- 视频解码：
  - 软件解码（CPU）
  - 硬解（D3D11VA）
- 输入回传：
  - 将窗口消息映射为 DataChannel 输入 JSON，上行到 Agent 执行注入。
- 退出行为：
  - 收到 `remoteProcessExited` 后退出并关闭窗口。

## 6. 端到端业务流程（时序与边界）

## 6.1 远程画面会话（浏览器）
- 前端连接信令 WebSocket。
- 前端发送 `request`（可含 exePath）。
- Agent 创建会话并发起本地 WebRTC 协商，发送 Offer。
- 前端收到 Offer，生成 Answer 并回发。
- WebRTC 建链完成后：
  - 视频轨开始出画。
  - DataChannel 打开后前端发送 `controlRequest`，Agent 返回 `controlGranted/Denied`。
  - 控制权获得后前端上行输入事件。

## 6.2 窗口缺失与恢复
- 采集线程在无可采集 surfaces 时进入 grace：
  - 进程仍存活：广播 `remoteWindowMissing`，前端仅提示等待，画面页不关闭。
  - 进程不存活：触发退出逻辑并广播 `remoteProcessExited`。

## 6.3 远端进程退出
- Agent 发现会话不存活或主动 stop 时：
  - 广播 `remoteProcessExited`。
  - 停止媒体采集与编码，解绑输入目标，释放会话资源。
- 前端/独立接收端收到后关闭画面页或退出程序。

## 6.4 断线与重连
- operator 连接丢失：
  - 进入宽限期等待重连；同一 client + 同一 exe 的请求可走 reconnect（不重启进程）。
  - 宽限期到期 teardown。

## 7. 对外协议（接口说明）

## 7.1 信令协议（WebSocket 文本 JSON）

### 7.1.1 前端/接收端 -> Agent（经信令中继）
- **媒体会话请求**
  - `{"id":"server","type":"request","exePath":"C:\\\\path\\\\app.exe"}`
- **仅文件会话请求**
  - `{"id":"server","type":"file_request"}`
- **SDP Answer**
  - `{"id":"server","type":"answer","sdp":"..."}`
- **停止会话**
  - `{"id":"server","type":"stop"}`

### 7.1.2 Agent -> 前端/接收端
- **SDP Offer（以及 ICE 候选等）**
  - `{"id":"<clientId>","type":"offer","sdp":"..."}`

说明：
- 信令服务按 `id` 转发，并在转发时把 `id` 改写为来源端 id（用于标识发送者）。

---

## 7.2 DataChannel 协议（文本：纯文本心跳 + JSON）

### 7.2.1 心跳（纯文本）
- Agent -> 前端：`Ping`
- 前端 -> Agent：`Pong <timestamp>`

约束：
- 双方均只对纯文本 `Ping/Pong` 做特判处理，避免误吞 JSON/base64 分片内容。

### 7.2.2 控制权
- 请求：`{"type":"controlRequest"}`
- 释放：`{"type":"controlRelease"}`
- 回包：
  - `{"type":"controlGranted"}`
  - `{"type":"controlDenied"}`
  - `{"type":"controlRevoked"}`

### 7.2.3 输入事件（前端/接收端 -> Agent）
- 鼠标移动：
  - `{"type":"mouseMove","x":dx,"y":dy,"absoluteX":ax,"absoluteY":ay,"videoWidth":vw,"videoHeight":vh}`
- 鼠标按键：
  - `{"type":"mouseDown","button":0|1|2,"x":0,"y":0}`
  - `{"type":"mouseUp","button":0|1|2,"x":0,"y":0}`
  - `{"type":"mouseDoubleClick","button":0|1|2,"x":0,"y":0}`
- 滚轮：
  - `{"type":"mouseWheel","deltaX":dx,"deltaY":dy,"x":0,"y":0}`
- 键盘：
  - `{"type":"keyDown","vk":<int>,"keyCode":<int>,"shiftKey":0|1,"ctrlKey":0|1,"altKey":0|1,"metaKey":0|1}`
  - `{"type":"keyUp", ... 同上 ...}`

说明：
- 非控制者输入会在 Agent 侧被丢弃（由控制权策略决定）。

### 7.2.4 关键帧请求（前端/接收端 -> Agent）
- `{"type":"requestKeyframe","reason":"...","detail":{...}}`

### 7.2.5 延迟对时（前端/接收端 <-> Agent）
- 请求：`{"type":"latPing","seq":N,"tCli":<ms>}`
- 回包：`{"type":"latPong","seq":N,"tCli":<ms>,"tSrv":<ms>}`

### 7.2.6 Agent 状态通知（Agent -> 前端/接收端）
- 远端进程退出：
  - `{"type":"remoteProcessExited"}`
- 远端窗口缺失（进程仍存活但当前无可采集 surfaces）：
  - `{"type":"remoteWindowMissing","why":"...","missingMs":5000}`
- 分辨率变化提示：
  - `{"type":"videoResolution","width":W,"height":H}`
- 采集后端健康摘要：
  - `{"type":"captureHealth","backend":"dxgi"|"gdi"}`

---

## 7.3 文件传输协议（DataChannel JSON）

### 7.3.1 目录列表
- 请求：`{"type":"fileList","path":"<relative>"}`
- 响应：`{"type":"fileListResult","path":"...","root":"<abs root>","entries":[...]}`  
  entries 字段示例：`{"name","path","isDir","size","mtime"}`

### 7.3.2 文本预览
- 请求：`{"type":"filePreview","path":"...","maxBytes":65536}`
- 响应（可预览）：
  - `{"type":"filePreviewResult","path":"...","previewable":true,"content":"..."}`
- 响应（不可预览）：
  - `{"type":"filePreviewResult","path":"...","previewable":false,"reason":"..."}`

### 7.3.3 下载
- init 请求：`{"type":"fileDownloadInit","path":"...","chunkSize":8192}`
- init 响应：`{"type":"fileDownloadReady","transferId":"...","fileName":"...","fileSize":N,"mtime":ms,"chunkSize":N}`
- chunk 请求：`{"type":"fileDownloadChunk","transferId":"...","offset":X,"size":N}`
- chunk 响应：`{"type":"fileDownloadChunkData","transferId":"...","offset":X,"size":N,"nextOffset":Y,"isEof":bool,"data":"<base64>"}`

### 7.3.4 上传
- init 请求：
  - `{"type":"fileUploadInit","requestId":"...","path":"<relative dir>","fileName":"a.bin","fileSize":N,"chunkSize":4096,"overwrite":true,"resume":false}`
- init 响应：
  - `{"type":"fileUploadReady","requestId":"...","transferId":"...","chunkSize":N,"nextOffset":X}`
- chunk 请求：
  - `{"type":"fileUploadChunk","transferId":"...","offset":X,"data":"<base64>"}`
- chunk ACK：
  - 成功：`{"type":"fileUploadAck","transferId":"...","ok":true,"uploadedBytes":Y,"isComplete":bool}`
  - 偏移不匹配：`{"type":"fileUploadAck","transferId":"...","ok":false,"expectedOffset":Y}`
- commit 请求：
  - `{"type":"fileUploadCommit","transferId":"..."}`
- commit 响应：
  - `{"type":"fileUploadCommitted","transferId":"...","path":"<relative final path>"}`

### 7.3.5 错误
- `{"type":"fileError","op":"fileUploadInit|fileUploadChunk|...","message":"...","transferId":"<optional>"}`

## 8. 配置项说明（运行时）

配置读取由 `runtime_config/runtime_settings` 等模块提供（Windows Agent 侧）。

### 8.1 信令
- `RPC_SIGNALING_IP`：信令服务器 IP（默认 `127.0.0.1`）
- `RPC_SIGNALING_PORT`：信令端口（默认 `9090`）

### 8.2 采集与编码
- `RPC_ACTIVE_FPS`：活跃帧率（默认 30；最小 1）
- `RPC_FILTER_CAPTURE_BLACK_FRAMES`：是否过滤疑似黑帧（默认 true）
- 采集后端/编码后端：由配置驱动的工厂创建（支持 DXGI/GDI 等）

### 8.3 文件服务
- `RPC_DATA_ROOT`：文件根目录（默认 `D:\rpc_data`）
- `RPC_FILE_CHUNK_SIZE`：建议分片大小（后端会做 4KB~64KB 钳制；前端也可自定义 chunkSize）

## 9. 日志与诊断

- Agent 侧关键日志标签（典型）：
  - `[signaling]`：信令连接与消息
  - `[session]`：会话创建/拆除、退出广播
  - `[capture]`：目标解析、surfaces 状态、抓帧失败
  - `[encode]`：编码布局/失败与关键帧请求
  - `[file]`：文件上传下载流程（若开启相关日志）
- 前端侧：
  - 控制台输出：延迟诊断、WebRTC stats、关键帧请求等
  - “我的数据”页面含传输日志与进度显示

---

## 10. 安全性约束（当前版本）

- **文件路径沙箱**：所有 file* 操作必须在 `RPC_DATA_ROOT` 下，禁止绝对路径与越界路径。
- **控制权模型**：非控制者输入事件丢弃；控制权授予策略由 Agent 侧回调决定。
- **信令层无鉴权**：当前 Qt 信令服务为简单中继，不做认证与访问控制；生产化需补充鉴权与访问控制策略。

---

## 11. 性能与稳定性策略（当前版本）

- 采集/编码与发送解耦：
  - “最新帧槽位 + 去重”避免积压，落后时直接追最新帧（天然丢帧）。
  - 编码样本队列为有界队列（默认最多 3 帧），溢出丢最旧，避免内存膨胀。
- 抖动/丢包恢复：
  - 前端/接收端可基于 stats 触发 `requestKeyframe`，Agent 侧请求编码器关键帧。
- 断线宽限：
  - operator 连接丢失后宽限期保活，超时 teardown。

## 12. 启动与使用说明（最小步骤）

### 12.1 启动信令服务
- 编译并运行 `signaling-server-qt`，默认监听：`ws://127.0.0.1:9090`

### 12.2 启动 Windows Agent
- Visual Studio 打开 `remote_process_control.sln`
- 启动项目：`remote_desktop_agent`
- 确认配置项 `RPC_SIGNALING_IP/PORT` 指向信令服务

### 12.3 打开前端
- 打开 `frontend/index.html`（可 `file://`）
- 在页面中配置/确认信令地址并发起连接
- 发起远程进程请求（选择或输入 exePath 触发 request）

## 13. 验收用例（建议）

### 13.1 远程画面
- 能启动指定 `exePath` 并在前端稳定出画（分辨率正确、无持续黑屏）。
- 在 10 分钟运行中无异常退出或资源明显泄漏（以任务管理器/日志观察）。

### 13.2 输入控制
- 获得控制权后，鼠标移动/点击/滚轮/键盘输入能作用于远端窗口。
- 分辨率变化/窗口位置变化后，鼠标映射仍基本正确（不出现明显偏移）。

### 13.3 文件管理
- 在 `RPC_DATA_ROOT` 下：
  - 列目录正确（包含文件与文件夹、mtime/size 合理）。
  - 文本文件可预览；非文本类型提示不可预览。
  - 下载小文件/大文件成功、内容一致。
  - 上传小文件/大文件成功；中断后可通过 offset 对齐机制继续（若前端启用重试/续传语义）。

### 13.4 异常路径
- 关闭远端进程后，前端能收到 `remoteProcessExited` 并退出画面页。
- 暂时无可采集窗口时，前端收到 `remoteWindowMissing` 并继续等待（不误判为退出）。
